# v3.0 GitHub Workflow Crisis - Lessons Learned

**Date:** 2026-03-24  
**Project:** pim-rl v3.0  
**Incident:** GitHub workflow crisis (18:38-19:15, ~37 minutes)  
**Impact:** Phase 3 → Phase 4 transition混乱，团队不知道Phase 4存在，Issue状态混乱  
**Root Cause:** 没有严格依赖GitHub Issues作为唯一真相来源

---

## Executive Summary

v3.0 Phase 3刚完成（4/4 issues closed），即将启动Phase 4时，团队发生严重的workflow crisis：

1. **不知道Phase 4存在** — 团队以为v3.0只有3个Phase
2. **Issues写完忘记关闭** — Issue #27, #28, #29, #30已完成但未在GitHub关闭
3. **Milestone错误** — Phase 3 issues未正确标记milestone
4. **进度判断错误** — 依赖本地文档而非GitHub Issues判断进度

**根本原因：** 团队没有养成"GitHub Issues = 唯一真相"的习惯，导致工作和Issues脱节。

**影响时间：** ~37分钟（18:38发现问题 → 19:15解决）

**教训：** 必须强制执行"Every decision, every task starts with `gh issue list`"原则。

## Root Cause Analysis (5 Whys)

### Why 1: 为什么团队不知道Phase 4存在？

**答：** 因为没有查看GitHub Issues (`gh issue list --milestone v3.0-phase4`)

### Why 2: 为什么没有查看GitHub Issues？

**答：** 因为习惯依赖本地文档（docs/v3.0/）判断进度

### Why 3: 为什么依赖本地文档而非GitHub？

**答：** 因为没有养成"GitHub = 唯一真相"的工作习惯

### Why 4: 为什么没有养成这个习惯？

**答：** 因为没有强制流程要求（没有enforcement mechanism）

### Why 5: 为什么没有enforcement？

**答：** 因为团队从未经历过"Issues和工作脱节"的严重后果，直到今天

**真正的Root Cause:**  
没有建立**强制性**的GitHub-first workflow，导致工作（本地代码+文档）和项目管理（GitHub Issues）两层皮。

## Specific Failure Modes

### 失败模式1：完成不关Issue

**现象：**
- Issue #27 (19:41 code完成 + 19:46 小A验收) → 未关闭
- Issue #28 (下午完成) → 未关闭
- Completion report写了，tests通过了，commit推了 → **但Issue还是OPEN**

**Why it happened:**
- 团队focus在"代码完成"，忽略"Issue关闭"
- 没有checklist强制要求"close issue"
- 验收流程不包括"verify GitHub status"

**Impact:**
- GitHub显示"4/4 Phase 3 issues OPEN" → 看起来0%完成
- 实际Phase 3 100%完成 → **严重不一致**
- YZ无法通过GitHub准确判断进度

**Should have done:**
```bash
# After code completion
pytest tests/test_multi_mode_ic.py  # 验证
git push origin main                # 提交
gh issue close 27 -c "Completed"    # 关闭Issue ← 这一步被跳过了！
```

## Actionable Fixes

### Fix 1: Enforce "GitHub First" Workflow

**Rule:**  
**每个决策/任务/讨论必须从`gh issue list`开始**

**Implementation:**

**Daily standup（每天早上）:**
```bash
# ∞负责运行并发到Discord
gh issue list --state open --json number,title,assignee,milestone | \
  jq -r '.[] | "\(.number): \(.title) (@\(.assignee.login), \(.milestone.title))"'
```

**Before starting work（开工前）:**
```bash
# 每个人运行（自己负责）
gh issue list --assignee @me --state open
# 确认今天要做的issue
```

**After completing task（完成后）:**
```bash
# 立即关闭issue（不等验收）
gh issue close <number> -c "Completed: [summary]"
# 如果需要验收，先close再reopen（保持GitHub状态准确）
```

**Weekly review（每周五）:**
```bash
# ∞负责检查
gh issue list --state all --json number,state,closedAt | \
  jq '.[] | select(.state == "CLOSED" and .closedAt == null)'
# 发现inconsistency → 立即修复
```

**Enforcement:**
- ∞每天检查：是否所有人都follow了workflow
- 违反者：记录到MEMORY.md，公开提醒
- 连续违反3次：escalate to YZ

### Fix 2: Issue Lifecycle Mandatory Steps

**Workflow:**

```
1. Create Issue
   ↓
2. Verify in GitHub (gh issue view <number>)
   ↓
3. Assign milestone + assignee
   ↓
4. Start work (local branch: issue-<number>-description)
   ↓
5. Complete code + tests
   ↓
6. Write completion report (docs/v3.0/issue<number>-completion.md)
   ↓
7. Push commit
   ↓
8. ✅ CLOSE ISSUE (gh issue close <number>)  ← 强制步骤！
   ↓
9. Verify closed (gh issue list --state closed | grep <number>)
```

**Checklist（必须100%完成）:**
- [ ] Code written
- [ ] Tests passing
- [ ] Completion report written
- [ ] Commit pushed
- [ ] **Issue closed in GitHub** ← 最容易忘的一步
- [ ] **Verified in `gh issue list`** ← 二次确认

**Automation (future):**
```bash
# Git hook: 检测commit message是否提到issue
# 如果包含"Close #27"，自动verify issue是否真的关闭
```

## Enforcement Mechanism

### Level 1: Self-Check (每个人负责)

**Daily routine:**
```bash
# Morning (9:00)
gh issue list --assignee @me --state open
# "今天我要做issue #X"

# Before work
gh issue view <number>
# 确认requirements

# After work
gh issue close <number>
gh issue list --assignee @me --state closed --limit 1
# 验证关闭成功
```

### Level 2: Peer Review (同事互查)

**每次验收时：**
- 验收人：检查GitHub issue状态
- 如果issue未关闭 → 拒绝验收
- 完成者必须先close issue才能获得验收

**Example (小A验收小P的Issue #27):**
```
小A: "代码看起来不错，但issue还没关。请先close issue。"
小P: (运行gh issue close 27)
小A: (验证) "好了，现在验收通过。"
```

### Level 3: ∞ Daily Check (系统监督)

**∞每天18:00运行:**
```bash
# 检查今天关闭的issues是否有completion report
gh issue list --state closed --json number,closedAt | \
  jq -r '.[] | select(.closedAt | startswith("2026-03-24")) | .number' | \
  while read num; do
    if [ ! -f "docs/v3.0/issue${num}-completion.md" ]; then
      echo "Warning: Issue #${num} closed without completion report"
    fi
  done
```

**发现问题 → Discord提醒 → 责任人24小时内修复**
